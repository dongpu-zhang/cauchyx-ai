① Planner Agent — 任务规划
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

职责：每周读取长期记忆，生成本周 RegScanner 扫描任务清单

──── 输入接口 ────────────────────────────────────────────────────────────────

触发方式
  · OPC Cron 每周一 06:00 UTC HTTP POST 调用
  · 或专家在控制台点击"立即触发"（手动 POST）

输入数据结构（JSON）
  {
    "trigger": "cron" | "manual",
    "expert_id": "expert_uuid",
    "service_id": "regwatch_001",
    "config": {
      "states": ["CA", "TX"],           // 监控州列表
      "sources": [                       // 监控来源配置
        {"name": "EPA", "url": "https://www.epa.gov/newsroom", "type": "rss"},
        {"name": "CalRecycle", "url": "https://calrecycle.ca.gov/rss", "type": "rss"},
        {"name": "CDPH", "url": "...", "type": "html"},
        {"name": "TCEQ", "url": "...", "type": "html"}
      ],
      "keywords": ["restaurant","food service","grease","wastewater","organic waste"],
      "scan_depth_days": 14             // 检查过去 14 天的发布内容
    }
  }

从记忆读取（ChromaDB / Mem0）
  · 上周已发现的法规 ID 列表（去重用）
  · 各来源上次成功扫描时间戳
  · 本月已推送法规摘要（避免重复预警）

──── 第三方方案评估 ────────────────────────────────────────────────────────────

可直接使用的框架
  · LangGraph（推荐）：用 StateGraph 定义 Planner 节点，内置状态持久化
    - 优点：天然支持 Think→Do→Check→Improve 循环，有记忆接口
    - 接入方式：pip install langgraph，定义 PlannerNode
  · CrewAI：定义 Planner 角色和 Goal，自动分解任务
    - 优点：内置角色系统，适合多 Agent 协作
    - 缺点：定制性较 LangGraph 差

需要定制的部分
  · RegWatch 专属 Prompt（中英双语，理解合规场景）
  · 记忆 Schema（法规 ID 格式、来源优先级）
  · 增量扫描逻辑（last_seen_id 追踪）

状态：🟡 轻度定制 — 1 周内可完成

──── 第三方方案评估 ────────────────────────────────────────────────────────────

可直接使用（推荐方案）
  · Firecrawl（firecrawl.dev）🟢
    - 功能：专业网页抓取，自动处理 JS 渲染、反爬、内容结构化
    - 接入：POST https://api.firecrawl.dev/v1/scrape，传 URL 返回 Markdown
    - 价格：$16/月 starter（足够 RegWatch 规模）
    - 适合：HTML 类政府网站（CDPH、TCEQ）

  · Apify（apify.com）🟢
    - 功能：有预制的 EPA.gov、FDA.gov 爬虫（Actor）
    - 接入：Apify Client SDK，调用现成 Actor
    - 优势：有 EPA RSS 解析 Actor，直接获取结构化法规数据
    - 价格：按计算单元收费，RegWatch 规模约 $10/月

  · feedparser（Python 库）🟢 免费
    - 适合：有 RSS/Atom feed 的来源（EPA、CalRecycle 均有官方 RSS）
    - 接入：pip install feedparser，5 行代码解析

  · pypdf2 / pdfplumber（Python 库）🟢 免费
    - 适合：PDF 格式的法规原文解析

需要定制的部分
  · 法规变更 Diff 检测（对比 last_seen_id，识别"UPDATED"状态）
  · 多来源结果合并去重（同一法规可能在多个来源出现）
  · 错误重试逻辑（政府网站偶发 503）

状态：🟡 轻度定制 — 1 周（Firecrawl + feedparser + 自定义 Diff 逻辑）

  推荐组合：
    RSS 来源（EPA/CalRecycle）→ feedparser（免费）
    HTML 来源（CDPH/TCEQ）   → Firecrawl API（$16/月）
    PDF 附件                  → pdfplumber（免费）

──── 第三方方案评估 ────────────────────────────────────────────────────────────

MVP 阶段（可直接使用）
  · Claude API（claude-sonnet-4-6）🟡
    - 接入：Anthropic SDK，精心设计 System Prompt
    - System Prompt 要点：角色设定（15年合规顾问）+ 输出 JSON Schema + 示例
    - 优点：零开发，当天可用，质量够用
    - 缺点：没有专家个人风格，解读可能过于保守
    - 成本：每次约 $0.01–$0.05（视法规长度）

  · GPT-4o（OpenAI）🟡
    - 类似，JSON mode 输出更稳定

生产阶段（需要开发）🔴
  · 专家微调模型（通过 OPC FTaaS 或 Together AI）
    - 训练数据：专家过去 5 年的 300 份法规解读文档
    - 微调方式：LoRA（低成本），基底模型建议 Llama-3-8B 或 Qwen-7B
    - 训练平台选项：
      ① OPC FTaaS（本平台）          — 成本最低，与平台深度集成
      ② Together AI Fine-tuning       — $0.1/K tokens，API 简单
      ③ Replicate Fine-tuning         — 支持私有模型托管
    - 开发工作：数据清洗 + 格式化（约 2 周）+ 训练（约 1 周）
    - 效果：解读风格与专家一致，减少 Critic 打回率

状态：MVP=🟡（Claude API 直接用）/ 生产=🔴（微调，3 周）

──── 内部 SQL 查询（PostgreSQL）─────────────────────────────────────────────

  -- 示例查询
  SELECT
    r.restaurant_id,
    r.name,
    r.email,
    r.phone,
    r.city,
    r.state,
    r.outdoor_seating,
    r.outdoor_seating_sqft,
    r.has_outdoor_washing,
    r.subscription_status,
    CASE
      WHEN r.outdoor_seating = true AND r.outdoor_seating_sqft > 0 THEN 'HIGH'
      WHEN r.has_outdoor_washing = true THEN 'HIGH'
      ELSE 'MEDIUM'
    END AS risk_level
  FROM restaurants r
  WHERE r.expert_id = $1
    AND r.subscription_status = 'active'
    AND r.state = ANY($2)
    AND (r.outdoor_seating = true OR r.has_outdoor_washing = true)
  ORDER BY risk_level DESC, r.name;

──── 餐馆档案 Schema（PostgreSQL 表）────────────────────────────────────────

  restaurants (
    restaurant_id     UUID PRIMARY KEY,
    expert_id         UUID REFERENCES experts(id),
    name              VARCHAR(200),
    email             VARCHAR(200),
    phone             VARCHAR(20),
    address           TEXT,
    city              VARCHAR(100),
    state             CHAR(2),         -- CA, TX, NY...
    zip               VARCHAR(10),
    county            VARCHAR(100),    -- 用于县级法规匹配

    -- 经营属性（法规匹配核心字段）
    business_type     VARCHAR(50),     -- full_service / fast_food / food_truck / catering
    seating_capacity  INT,
    outdoor_seating   BOOLEAN,
    outdoor_seating_sqft INT,
    has_outdoor_washing  BOOLEAN,
    has_parking_lot   BOOLEAN,
    has_grease_trap   BOOLEAN,         -- 已安装油脂截留器
    serves_alcohol    BOOLEAN,
    has_drive_through BOOLEAN,

    -- 合规状态
    compliance_status JSONB,           -- {"EPA-40CFR453": "compliant", ...}

    -- 订阅信息
    subscription_status VARCHAR(20),   -- active / paused / cancelled
    subscribed_at     TIMESTAMP,
    language_pref     VARCHAR(5),      -- en / zh / es

    created_at        TIMESTAMP,
    updated_at        TIMESTAMP
  )

──── 第三方方案评估 ────────────────────────────────────────────────────────────

部分可用的第三方工具
  · LangChain SQL Agent 🟡
    - 可以用自然语言生成 SQL（"找到有室外座位的加州餐馆"）
    - 适合 MVP 阶段：快速搭建，但生成 SQL 不稳定
    - 生产阶段不推荐：SQL 注入风险，性能差

  · Directus / Hasura（GraphQL API over PostgreSQL）🟡
    - 自动生成 CRUD API，RiskMapper 只需调 GraphQL
    - 节省约 1 周后端开发时间

必须自己开发的部分 🔴
  · 餐馆属性 Schema 设计（业务核心，无法外包）
  · 法规类型 → 匹配条件的规则引擎
    例：
    "has_outdoor_seating" 法规 → 查 outdoor_seating=true 的餐馆
    "grease_trap_required" 法规 → 查 has_grease_trap=false 的餐馆
    "county_specific" 法规 → 按 county 字段过滤
  · 已合规状态追踪（避免重复预警合规餐馆）
  · 风险等级计算逻辑

状态：🔴 需要开发 — 规则引擎 + Schema 设计约 2 周

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

职责：对 AlertComposer 输出的预警进行质量评分，不合格打回重写

──── 输入接口 ────────────────────────────────────────────────────────────────

  {
    "critic_job_id": "critic_uuid",
    "emails": [...],                   // AlertComposer 的全部输出
    "evaluation_rubric": {            // 评分标准（专家在 OPC 控制台配置）
      "required_fields": [
        "violation_amount",           // 必须包含违规金额
        "deadline_date",              // 必须包含截止日期
        "action_steps_count_min": 3,  // 至少 3 步整改建议
        "personalization_check": true // 必须包含餐馆名
      ],
      "style_checks": [
        "no_legal_jargon",            // 不能出现法律术语（如 "promulgated"）
        "plain_english_grade_8"       // 可读性 ≤ 8 年级水平
      ],
      "pass_threshold": 8.0
    },
    "sample_size": 10                 // 随机抽样 10 封精细评审，其余规则检查
  }

──── 评分算法 ────────────────────────────────────────────────────────────────

  规则检查（所有邮件，自动）
    · 包含金额关键词（$X,XXX）：+2 分
    · 包含日期（YYYY-MM-DD 格式）：+2 分
    · 包含 3 个以上 "Step N" 标记：+2 分
    · 包含餐馆名称：+1 分

  LLM 评审（抽样 10 封，Claude-as-Judge）
    · Prompt："请从餐馆老板视角评估这封预警邮件是否清晰、可执行、有说服力。
               评分 0-10，给出理由和具体改进建议。"
    · 输出：{ "score": 8.5, "issues": ["第2条整改建议不够具体"], "suggestions": [...] }

──── 三类记忆存储 ────────────────────────────────────────────────────────────

类型 A：法规向量库（ChromaDB Collection: regulations）

  Schema:
  {
    "id": "EPA-40CFR453-2026-06-05",
    "embedding": [...],               // 法规全文的 embedding
    "metadata": {
      "title": "...",
      "effective_date": "2026-09-05",
      "source": "EPA",
      "affected_types": ["has_outdoor_seating"],
      "penalty_max": 10000,
      "sent_count": 49,               // 已推送给多少餐馆
      "discovered_at": "2026-06-08"
    }
  }

类型 B：餐馆合规时间线（PostgreSQL: restaurant_compliance_log）
  restaurant_compliance_log (
    log_id          UUID,
    restaurant_id   UUID,
    reg_id          VARCHAR,
    action          VARCHAR,    -- "alerted" / "complied" / "pending" / "waived"
    action_date     DATE,
    channel         VARCHAR,    -- "email" / "sms"
    opened          BOOLEAN,    -- 邮件是否已读
    replied         BOOLEAN,
    notes           TEXT
  )

类型 C：风格偏好（ChromaDB Collection: restaurant_styles）
  存储：该餐馆历史邮件的语气、回复倾向，供 AlertComposer 个性化

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

所有 Agent 的调度由编排层统一管理

状态机节点（LangGraph StateGraph）
  START
    → planner_node          （生成扫描任务）
    → reg_scanner_node      （抓取法规，并行 4 个来源）
    → legal_interpreter_node（解读法规）
    → risk_mapper_node      （匹配餐馆）
    → alert_composer_node   （生成预警）
    → critic_node           （质量评审）
    → [条件分支]
        if critic.result == "APPROVED" → expert_review_node
        if critic.result == "REVISION" and rounds < 3 → alert_composer_node（重试）
        if critic.result == "REVISION" and rounds >= 3 → expert_review_node（带标注）
    → expert_review_node    （等待专家批准，最长 24h）
    → dispatcher_node       （发送预警）
    → memory_update_node    （更新记忆）
    → END

  QA Agent 作为独立服务常驻运行，不进入主状态机

状态结构（LangGraph State）
  {
    "session_id": str,
    "week": str,
    "config": ExpertConfig,
    "scan_tasks": list[ScanTask],
    "findings": list[Regulation],
    "interpretations": list[Interpretation],
    "affected_restaurants": list[AffectedRestaurant],
    "composed_emails": list[Email],
    "critic_result": CriticResult,
    "revision_round": int,
    "expert_approved": bool,
    "dispatch_result": DispatchResult,
    "errors": list[str]
  }

第三方方案
  · LangGraph（langchain-ai/langgraph）🟡 — 推荐
  · n8n（n8n.io）🟡 — 可视化工作流，适合非开发者配置（但复杂逻辑受限）
  · Prefect / Airflow 🟡 — 纯数据管道场景，不适合 LLM Agent 循环

状态：🟡 轻度定制 — 1–2 周（状态机定义 + 节点连接 + 错误处理）
